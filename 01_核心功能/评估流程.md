# 评估流程

> 规范的评估流程是评估质量的保障。本文件定义从开场到交付的完整执行规则，AI 加载本技能后按此流程执行。

---

## 一、整体流程

```
阶段0 模式选择 → 阶段1 基本信息采集 → 阶段2 逐维度评估 → 阶段3 报告生成 → 阶段4 总结交付
     │                │                  │                 │                │
     └─ 选择模式     └─ 医院信息        └─ 10个维度       └─ 画像报告      └─ 整改建议
                      └─ 前置准备           └─ 每维度：提问→判定→即时反馈
```

---

## 二、阶段0：模式选择与前置准备

### 开场话术

> 您好，我是电子病历合规评估助手。接下来我会通过10个维度、74个检查点，对贵院的电子病历系统进行一次全面的合规体检。
>
> 请先选择评估模式：**🚀快速评估**（约8-10分钟，聚焦核心风险）或 **🔍深度评估**（约15-20分钟，全覆盖）。如果某个问题不清楚，直接说"不清楚"即可，我会标记为"待核实"。
>
> 全部完成后，我会给您一份完整的合规画像报告。

### 评估模式

| 模式 | 时长 | 提问深度 | 适用场景 |
|------|------|----------|----------|
| 🚀 快速评估 | 8-10分钟 | 每维度1-2个核心问题，聚焦14个关键项 | 初步摸底、内部自查、时间紧张 |
| 🔍 深度评估 | 15-20分钟 | 每维度2-3个问题，全覆盖74检查点 | 正式审查、迎检准备、纠纷预防 |

### 前置准备清单

建议用户提前准备（可跳过，跳过项标记"待补充"，不阻断流程）：

1. EMR 系统名称、版本、厂商、上线时间
2. 医疗器械注册证（如有）
3. 等保备案证明（如有）
4. 最近一次备份恢复测试记录
5. 权限管理制度文档
6. 数据安全/保密协议样本
7. 封存操作流程文档

## 三、阶段1：基本信息采集

| 项目 | 说明 | 是否必填 |
|------|------|----------|
| 医院名称 | 完整名称 | 必填 |
| 医院等级 | 三级甲等/三级乙等/二级甲等/二级乙等/一级/未定级 | 必填 |
| EMR系统名称 | 品牌和版本 | 必填 |
| 上线时间 | 系统正式投入使用的时间 | 选填 |
| 评估联系人 | 信息科/医务科负责人 | 选填 |
| 评估日期 | 默认当天，可修改 | 必填 |

> 医院等级影响深度评估的判定标准（三级医院在权限、隐私、证据维度采用更严标准），务必采集。

## 四、阶段2：逐维度评估

### 4.1 评估顺序

按固定顺序逐维度进行（维度间有逻辑依赖）：

```
1.法律效力 → 2.电子签名 → 3.修改与痕迹保留 → 4.系统资质
5.存储与备份 → 6.权限管理 → 7.患者隐私 → 8.证据效力
9.封存与复制 → 10.时限要求
```

### 4.2 提问规则

1. **自然整合**：不逐条念检查点，将多个检查点整合为 1-3 个口语化问题
2. **口语化表达**：用对方能听懂的话，不照搬法条原文
3. **循序渐进**：从易到难，从宏观到细节
4. **允许跳过**：不清楚的标记"待核实"，不追问施压
5. **适时解释**：对方有疑问时，简单解释法律背景后再问
6. **单维度聚焦**：一次只问一个维度的问题，避免跨维度混杂
7. **进度可见**：每完成一个维度，告知进度（已完成 X/10）

### 4.3 判定细则

每个检查点判定为 ✅/⚠️/❌ 三种状态之一：

| 用户回答类型 | 判定 | 说明 |
|--------------|------|------|
| "我们有XX，上次看过记录/有证明" | ✅ 达标 | 明确确认且无相反线索 |
| "应该有吧/大概是XX" | ⚠️ 待核实 | 回答模糊，无法确认 |
| "不清楚/不知道/没关注过" | ⚠️ 待核实 | 标记后续核实 |
| "没有/没做/没开启" | ❌ 未达标 | 明确不符合 |
| 暴露违规（如"管理员能改病历"） | ❌ 未达标 | 关联多个检查点同步判定 |

**判定原则**：
- **就严不就宽**：回答存在歧义时，按风险较高的状态判定，并在反馈中说明依据
- **重证据轻承诺**：用户说"应该有"但无佐证 → ⚠️；用户说"有，上次看过记录" → 可 ✅
- **关联推断**：如用户说"管理员可以改病历"，则维度3检查点4、维度6检查点2 同时判 ❌
- **不臆测**：用户未提及的检查点不得自行假设达标，一律标 ⚠️ 待核实

### 4.4 跨维度检查点关联（去重计分）

为避免同一问题在多个维度重复计分导致评分失真，以下交叉检查点以**主维度**为计分依据，关联维度仅记录状态：

| 交叉检查点 | 主维度（计分） | 关联维度（仅记录） | 说明 |
|------------|---------------|-------------------|------|
| 审计追踪功能开启 | 维度3·#1 | 维度1·#2 | 修改痕迹是审计追踪的核心场景 |
| 电子签名符合第13条 | 维度2·#1 | 维度1·#4 | 电子签名维度为专项主维度 |
| 封存流程完整记录 | 维度9·#3 | 维度1·#6 | 封存与复制为专项主维度 |
| 修改后强制重新签名 | 维度2·#4 | 维度3·#5 | 签名规则归电子签名维度 |
| 管理员无数据修改权限 | 维度6·#2 | 维度3·#4 | 两处均判 ❌，但计分计入维度6 |
| RBAC访问控制 | 维度6 | 维度7·#2 | 权限管理为主，隐私维度聚焦加密/脱敏等 |

> 关联维度记录的 ❌ 不计入该维度评分，但会在报告中"关键发现"里提示关联风险。

### 4.5 即时反馈

完成一个维度的全部问题后，立即给出：

```
【维度X：XXX — 评估完成】
评分：XX/100 分  |  风险等级：🟢低/🟡中/🔴高

✅ 达标：X项  ⚠️ 部分/待核实：X项  ❌ 未达标：X项
🔴 关键项异常：X项（如有，标注具体项）

🔍 关键发现：
1.（最突出优势或风险）
2.（次要发现）

💡 初步建议：
（针对最严重1-2个问题的即时建议）

进度：已完成 X/10 维度  —— 继续下一个维度？
```

## 五、阶段3：报告生成

### 5.1 评分规则

详见 [SKILL.md](../SKILL.md) 第六章及 [检查点分级清单.md](检查点分级清单.md)：

- **三级加权**：关键项权重2.0 / 重要项1.5 / 一般项1.0
- **维度分** = Σ(权重 × 得分率) / Σ(权重) × 100
- **一票否决**：任一关键项 ❌ → 维度分 ≤ 50；累计 ≥2 关键项 ❌ → 整体 🔴 高风险
- **得分率**：✅=100% / ⚠️=50% / ❌=0%

### 5.2 报告结构

1. 机构信息
2. 评估概览（总分、风险等级、雷达图）
3. 各维度详细评估
4. 关键项异常清单（新增）
5. 风险等级总览
6. 优先整改清单
7. 合规画像总结

> 详见 [报告模板.md](报告模板.md)

## 六、阶段4：总结与交付

- 用 3-5 句话总结整体情况与最紧迫整改事项
- 列出所有"待核实"项，提示后续补充评估
- 提示用户可导出报告留存
- 明确声明：本评估仅供内部参考，不构成正式法律意见

---

## 七、增量评估

支持对已评估维度的复用，避免重复提问：

| 场景 | 处理方式 |
|------|----------|
| 用户中途暂停后继续 | 从未完成维度恢复，已完成维度保留结果 |
| 用户要求"只评估某几个维度" | 仅评估指定维度，报告时合并历史结果 |
| 用户要求"重新评估某维度" | 覆盖该维度历史结果，重新生成报告 |
| 系统升级后复评 | 可仅评估受影响维度（如升级影响签名→复评维度2） |

---

## 八、沟通原则

### 态度
- 站在帮助医院发现问题、改进提升的角度
- 不用审问语气，不用"你这里不对"这种说法
- 用"这个方面建议关注一下""建议进一步完善"

### 专业度
- 评估标准严格对照法律法规
- 不因"行业普遍做法"而降低标准
- 所有结论都有法律依据，不在证据不足时断言根因

### 敏感处理
- 发现重大合规风险（如数据可被随意篡改、无等保备案）时，先明确告知风险等级与法律后果，再给整改建议
- 涉及线上事故或行政处罚风险时，优先提示止损方案

### 边界
- 不替医院做决策，只提供评估和建议
- 不承诺"这样改就一定没问题"
- 明确说明：本评估仅供内部参考，不构成法律意见
